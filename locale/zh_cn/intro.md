## 摘要

本文档介绍 RISC-V 指令集架构的基本概念和设计原理。本文涵盖了RISC-V的设计目标、硬件平台术语、软件执行环境、指令集概述、内存模型以及异常处理等核心内容。

### 涉及的指令格式

本文档主要涉及指令编码的基础格式，具体指令格式将在各扩展文档中详细说明：

#### 基础32位指令格式
```
标准32位指令: [31:2] 指令位 | [1:0] = 11
压缩16位指令: [15:2] 指令位 | [1:0] = 00/01/10
```

#### 指令边界对齐
- **IALIGN**: 指令地址对齐约束（16或32位）
- **ILEN**: 实现支持的最大指令长度

## 简介

RISC-V（读作"risk-five"）是一个新的指令集架构（ISA），最初设计用于支持计算机架构研究和教育，但我们现在希望它也能成为工业实现的标准、自由和开放的架构。我们在定义 RISC-V 时的目标包括：

* 一个完全**开放**的 ISA，对学术界和工业界完全免费开放。
* 一个**真实**的 ISA，适合直接本机硬件实现，而不仅仅是模拟或二进制翻译。
* 一个 ISA，避免为特定微架构风格（例如，微码、顺序、解耦、乱序）或实现技术（例如，全定制、ASIC、FPGA）进行"过度架构"，但允许在任何这些方面进行高效实现。
* 一个分离为**小型**基础整数 ISA 的 ISA，可单独作为定制加速器或教育用途的基础，以及可选的标准扩展，以支持通用软件开发。
* 支持修订的 2008 IEEE-754 浮点标准。
* 一个支持广泛 ISA 扩展和专用变体的 ISA。
* 应用程序、操作系统内核和硬件实现的 32 位和 64 位地址空间变体。
* 一个支持高度并行多核或众核实现的 ISA，包括异构多处理器。
* 可选的**变长指令**，既扩展了可用的指令编码空间，又支持可选的**密集指令编码**，以改善性能、静态代码大小和能效。
* 一个完全可虚拟化的 ISA，以简化虚拟机监控程序开发。
* 一个简化新特权架构设计实验的 ISA。

> **注释：**
> 对我们设计决策的评论格式如本段所示。
> 这种非规范性文本如果读者只对规范本身感兴趣可以跳过。

> **注释：**
> RISC-V 这个名字被选择来代表来自加州大学伯克利分校的第五个主要 RISC ISA 设计
> （RISC-I、RISC-II、SOAR 和 SPUR 是前四个）。我们还使用罗马数字"V"来表示
> "变体"和"向量"的双关语，因为对一系列架构研究的支持，包括各种数据并行加速器，
> 是 ISA 设计的明确目标。

RISC-V ISA 的定义尽可能避免实现细节（尽管包含了对实现驱动决策的评论），应被视为各种实现的软件可见接口，而不是特定硬件工件的设计。RISC-V 手册分为两卷。本卷涵盖基础**非特权**指令的设计，包括可选的非特权 ISA 扩展。非特权指令是那些通常在所有特权架构的所有特权模式中可用的指令，尽管行为可能因特权模式和特权架构而异。第二卷提供第一个（"经典"）特权架构的设计。手册使用 IEC 80000-13:2008 约定，一个字节为 8 位。

> **注释：**
> 在非特权 ISA 设计中，我们试图消除对特定微架构特性（如缓存行大小）或特权架构细节（如页面翻译）的任何依赖。这既是为了简单性，也是为了允许替代微架构或替代特权架构的最大灵活性。

## RISC-V 硬件平台术语

一个 RISC-V 硬件平台可以包含一个或多个 RISC-V 兼容处理核心以及其他非 RISC-V 兼容核心、固定功能加速器、各种物理内存结构、I/O 设备以及允许组件通信的互连结构。

如果一个组件包含独立的指令获取单元，则称为**核心**。一个 RISC-V 兼容核心可能通过多线程支持多个 RISC-V 兼容硬件线程，或**硬件线程（hart）**。

一个 RISC-V 核心可能有额外的专用指令集扩展或添加的**协处理器**。我们使用术语**协处理器**来指代连接到 RISC-V 核心的单元，主要由 RISC-V 指令流排序，但包含额外的架构状态和指令集扩展，并且可能相对于主 RISC-V 指令流具有一些有限的自主性。

我们使用术语**加速器**来指代不可编程的固定功能单元或可以自主运行但专门用于某些任务的核心。在 RISC-V 系统中，我们预期许多可编程加速器将是基于 RISC-V 的核心，具有专用指令集扩展和/或定制协处理器。RISC-V 加速器的一个重要类别是 I/O 加速器，它们从主应用核心卸载 I/O 处理任务。

RISC-V 硬件平台的系统级组织可以从单核微控制器到数千节点的共享内存众核服务器节点集群。即使是小型片上系统也可能被构造为多计算机和/或多处理器的层次结构，以模块化开发工作或在子系统之间提供安全隔离。

## RISC-V 软件执行环境和硬件线程

RISC-V 程序的行为取决于它运行的执行环境。RISC-V 执行环境接口（EEI）定义程序的初始状态、环境中硬件线程的数量和类型（包括硬件线程支持的特权模式）、内存和 I/O 区域的可访问性和属性、在每个硬件线程上执行的所有合法指令的行为（即，ISA 是 EEI 的一个组件）以及执行期间引发的任何中断或异常的处理，包括环境调用。EEI 的示例包括 Linux 应用程序二进制接口（ABI）或 RISC-V 监管者二进制接口（SBI）。RISC-V 执行环境的实现可以是纯硬件、纯软件或硬件和软件的组合。例如，操作码陷阱和软件仿真可用于实现硬件中未提供的功能。执行环境实现的示例包括：

* "裸机"硬件平台，其中硬件线程由物理处理器线程直接实现，指令完全访问物理地址空间。硬件平台定义了一个从上电复位开始的执行环境。
* RISC-V 操作系统，通过将用户级硬件线程复用到可用的物理处理器线程上并通过虚拟内存控制内存访问来提供多个用户级执行环境。
* RISC-V 虚拟机监控程序，为客户操作系统提供多个监管者级执行环境。
* RISC-V 仿真器，如 Spike、QEMU 或 rv8，在底层 x86 系统上仿真 RISC-V 硬件线程，可以提供用户级或监管者级执行环境。

> **注释：**
> 裸硬件平台可以被认为定义了一个 EEI，其中可访问的硬件线程、内存和其他设备填充环境，初始状态是上电复位时的状态。通常，大多数软件设计为使用硬件的更抽象接口，因为更抽象的 EEI 在不同硬件平台之间提供更大的可移植性。通常 EEI 彼此分层，其中一个更高级别的 EEI 使用另一个更低级别的 EEI。

从在给定执行环境中运行的软件的角度来看，硬件线程是一个在该执行环境内自主获取和执行 RISC-V 指令的资源。在这方面，硬件线程的行为就像硬件线程资源，即使被执行环境时间复用到真实硬件上。一些 EEI 支持创建和销毁额外的硬件线程，例如，通过环境调用来分叉新的硬件线程。

执行环境负责确保其每个硬件线程的最终前向进展。对于给定的硬件线程，当硬件线程正在行使明确等待事件的机制（如本规范第二卷中定义的等待中断指令）时，该责任被暂停；如果硬件线程被终止，该责任结束。以下事件构成前向进展：

* 指令的退休。
* 如 <<trap-defn>> 中定义的陷阱。
* 由扩展定义的任何其他构成前向进展的事件。

> **注释：**
> 术语硬件线程在 Lithe 的工作中引入，提供一个术语来表示抽象执行资源，而不是软件线程编程抽象。
>
> 硬件线程（hart）和软件线程上下文之间的重要区别是，在执行环境内运行的软件不负责导致其每个硬件线程的进展；这是外部执行环境的责任。因此，从执行环境内软件的角度来看，环境的硬件线程像硬件线程一样运行。
>
> 执行环境实现可能将一组客户硬件线程时间复用到其自己执行环境提供的较少主机硬件线程上，但必须以使客户硬件线程像独立硬件线程一样运行的方式进行。特别是，如果客户硬件线程多于主机硬件线程，则执行环境必须能够抢占客户硬件线程，并且不得无限期等待客户硬件线程上的客户软件"让出"客户硬件线程的控制。

## RISC-V ISA 概述

RISC-V ISA 被定义为基础整数 ISA（必须存在于任何实现中）加上基础 ISA 的可选扩展。基础整数 ISA 与早期 RISC 处理器非常相似，除了没有分支延迟槽并支持可选的变长指令编码。基础被仔细限制为足以为编译器、汇编器、链接器和操作系统（具有额外的特权操作）提供合理目标的最小指令集，因此提供了一个方便的 ISA 和软件工具链"骨架"，围绕它可以构建更多定制的处理器 ISA。

尽管方便地谈论 **RISC-V ISA**，RISC-V 实际上是一个相关 ISA 的家族，目前有四个基础 ISA。每个基础整数指令集的特征是整数寄存器的宽度和相应地址空间的大小以及整数寄存器的数量。有两个主要的基础整数变体，RV32I 和 RV64I，在第2章和第3章中描述，它们分别提供 32 位或 64 位地址空间。我们使用术语 XLEN 来指代整数寄存器的位宽（32 或 64）。第4章描述了 RV32I 或 RV64I 基础指令集的 RV32E 和 RV64E 子集变体，它们被添加来支持小型微控制器，并且具有一半数量的整数寄存器。基础整数指令集使用二进制补码表示有符号整数值。

> **注释：**
> 尽管 64 位地址空间是较大系统的要求，我们相信 32 位地址空间在未来几十年对于许多嵌入式和客户端设备仍然足够，并且希望降低内存流量和能耗。此外，32 位地址空间对于教育目的是足够的。更大的扁平 128 位地址空间最终可能是必需的，可以在现有 RISC-V ISA 框架内通过新的 RV128I 基础 ISA 来容纳。

> **注释：**
> RISC-V 中的四个基础 ISA 被视为不同的基础 ISA。一个常见的问题是为什么没有单一的 ISA，特别是为什么 RV32I 不是 RV64I 的严格子集？一些早期的 ISA 设计（SPARC、MIPS）在增加地址空间大小以支持在新的 64 位硬件上运行现有 32 位二进制文件时采用了严格超集政策。
>
> 显式分离基础 ISA 的主要优势是每个基础 ISA 可以根据其需求进行优化，而不需要支持其他基础 ISA 所需的所有操作。例如，RV64I 可以省略仅在 RV32I 中应对较窄寄存器所需的指令和 CSR。RV32I 变体可以使用仅由更宽地址空间变体所需的指令保留的编码空间。
>
> 不将设计视为单一 ISA 的主要缺点是它使在另一个上仿真一个基础 ISA 所需的硬件复杂化（例如，在 RV64I 上的 RV32I）。然而，寻址和非法指令陷阱的差异通常意味着在任何情况下即使具有完整超集指令编码也需要某种模式切换，并且不同的 RISC-V 基础 ISA 足够相似，支持多个版本的成本相对较低。

RISC-V 设计为支持广泛的定制和专业化。每个基础整数 ISA 可以用一个或多个可选指令集扩展进行扩展。扩展可以被分类为标准、定制或不符合标准。为此，我们将每个 RISC-V 指令集编码空间（和相关编码空间如 CSR）分为三个不相交的类别：**标准**、**保留**和**定制**。标准扩展和编码由 RISC-V International 定义；任何未由 RISC-V International 定义的扩展都是**非标准**的。每个基础 ISA 及其标准扩展仅使用标准编码，并且在使用这些编码时不应相互冲突。保留编码目前未定义，但为未来标准扩展保存；一旦如此使用，它们成为标准编码。定制编码永远不应用于标准扩展，并且可用于供应商特定的非标准扩展。非标准扩展是使用仅定制编码的定制扩展，或使用任何标准或保留编码的**不符合标准**扩展。指令集扩展通常是共享的，但可能根据基础 ISA 提供略有不同的功能。我们还开发了 RISC-V 基础指令和指令集扩展的命名约定，在命名章节中详细描述。

为了支持更通用的软件开发，定义了一组标准扩展来提供整数乘除、原子操作以及单精度和双精度浮点算术。基础整数 ISA 命名为"I"（根据整数寄存器宽度以 RV32 或 RV64 为前缀），包含整数计算指令、整数加载、整数存储和控制流指令。标准整数乘除扩展命名为"M"，添加了乘以和除以整数寄存器中保存的值的指令。标准原子指令扩展，表示为"A"，添加了原子读取、修改和写入内存的指令，用于处理器间同步。标准单精度浮点扩展，表示为"F"，添加了浮点寄存器、单精度计算指令以及单精度加载和存储。标准双精度浮点扩展，表示为"D"，扩展了浮点寄存器，并添加了双精度计算指令、加载和存储。标准"C"压缩指令扩展提供了常见指令的较窄 16 位形式。

除了基础整数 ISA 和这些标准扩展之外，我们相信新指令为所有应用程序提供显著好处是罕见的，尽管它可能对特定领域非常有益。随着能效关注迫使更大的专业化，我们相信简化 ISA 规范的必需部分是重要的。而其他架构通常将其 ISA 视为单一实体，随着时间推移添加指令而更改为新版本，RISC-V 将努力随时间保持基础和每个标准扩展不变，而是将新指令层叠为进一步的可选扩展。例如，基础整数 ISA 将继续作为完全支持的独立 ISA，无论任何后续扩展如何。

## 内存

RISC-V 硬件线程对所有内存访问都有一个单一的字节可寻址地址空间，大小为 2^XLEN 字节。内存的**字**定义为 32 位（4 字节）。相应地，**半字**是 16 位（2 字节），**双字**是 64 位（8 字节），**四字**是 128 位（16 字节）。内存地址空间是循环的，因此地址 2^XLEN-1 处的字节与地址零处的字节相邻。相应地，硬件完成的内存地址计算忽略溢出，而是绕回模 2^XLEN。

执行环境确定硬件资源到硬件线程地址空间的映射。硬件线程地址空间的不同地址范围可能（1）包含**主内存**，或（2）包含一个或多个 **I/O 设备**。I/O 设备的读取和写入可能有可见的副作用，但对主内存的访问不能。空置地址范围不是单独的类别，但可以表示为不可访问的主内存或 I/O 区域。尽管执行环境可能将硬件线程地址空间中的所有内容称为 I/O 设备，通常预期某些部分将被指定为主内存。

当 RISC-V 平台有多个硬件线程时，任何两个硬件线程的地址空间可能完全相同，或完全不同，或可能部分不同但共享一些资源子集，映射到相同或不同的地址范围。

> **注释：**
> 对于纯粹的"裸机"环境，所有硬件线程可能看到相同的地址空间，完全由物理地址访问。然而，当执行环境包括使用地址翻译的操作系统时，通常为每个硬件线程提供一个主要或完全属于自己的虚拟地址空间。

执行每个 RISC-V 机器指令需要一个或多个内存访问，细分为**隐式**和**显式**访问。对于执行的每个指令，完成**隐式**内存读取（指令获取）以获取要执行的编码指令。许多 RISC-V 指令除了指令获取之外不执行进一步的内存访问。特定的加载和存储指令在由指令确定的地址执行内存的**显式**读取或写入。执行环境可能规定指令执行执行其他**隐式**内存访问（如实现地址翻译）超出非特权 ISA 记录的那些。

执行环境确定地址空间的哪些部分可用于每种内存访问。例如，可以为指令获取隐式读取的位置集合可能与可以由加载指令显式读取的位置集合有任何重叠或没有重叠；可以由存储指令显式写入的位置集合可能只是可以读取的位置的子集。通常，如果指令尝试访问不可访问地址的内存，则为指令引发异常。

除非另有规定，不引发异常且没有副作用的隐式读取可能任意早且推测性地发生，甚至在机器可能证明需要读取之前。例如，有效实现可能尝试在最早机会读取所有主内存，缓存尽可能多的可获取（可执行）字节以供以后指令获取，并避免再次为指令获取读取主内存。为了确保某些隐式读取仅在对相同内存位置的写入之后排序，软件必须执行为此目的定义的特定栅栏或缓存控制指令（如 <<zifencei>> 中定义的 FENCE.I 指令）。

硬件线程进行的内存访问（隐式或显式）可能在另一个硬件线程或任何其他可以访问相同内存的代理感知下以不同顺序出现。然而，内存访问的这种感知重排序总是受到适用内存一致性模型的约束。RISC-V 的默认内存一致性模型是 RISC-V 弱内存排序（RVWMO），在内存模型章节和附录中定义。可选地，实现可能采用更强的全存储排序模型，如全存储排序章节中定义。执行环境也可能添加进一步限制内存访问感知重排序的约束。由于 RVWMO 模型是任何 RISC-V 实现允许的最弱模型，为此模型编写的软件与所有 RISC-V 实现的实际内存一致性规则兼容。与隐式读取一样，软件必须执行栅栏或缓存控制指令以确保内存访问的特定排序超出假定内存一致性模型和执行环境的要求。

## 基础指令长度编码

基础 RISC-V ISA 具有固定长度的 32 位指令，必须在 32 位边界上自然对齐。然而，标准 RISC-V 编码方案设计为支持具有可变长度指令的 ISA 扩展，其中每个指令可以是任意数量的 16 位指令**包裹**长度，包裹在 16 位边界上自然对齐。压缩章节中描述的标准压缩 ISA 扩展通过提供压缩的 16 位指令减少代码大小，并放松对齐约束以允许所有指令（16 位和 32 位）在任何 16 位边界上对齐以改善代码密度。

我们使用术语 IALIGN（以位为单位测量）来指代实现强制执行的指令地址对齐约束。IALIGN 在基础 ISA 中是 32 位，但一些 ISA 扩展，包括压缩 ISA 扩展，将 IALIGN 放松到 16 位。IALIGN 不能取除 16 或 32 以外的任何值。

我们使用术语 ILEN（以位为单位测量）来指代实现支持的最大指令长度，它总是 IALIGN 的倍数。对于仅支持基础指令集的实现，ILEN 是 32 位。支持更长指令的实现具有更大的 ILEN 值。

基础 ISA 中的所有 32 位指令的最低两位设置为 `11`。可选的压缩 16 位指令集扩展的最低两位等于 `00`、`01` 或 `10`。

> **注释：**
> 鉴于压缩格式的代码大小和能源节省，我们希望将压缩格式支持内置到 ISA 编码方案中，而不是将其作为事后思考添加，但为了允许更简单的实现，我们不希望使压缩格式成为强制性的。我们还希望可选地允许更长的指令以支持实验和更大的指令集扩展。尽管我们的编码约定需要对核心 RISC-V ISA 进行更严格的编码，但这有几个有益的效果。
>
> 标准 IMAFD ISA 的实现仅需要在指令缓存中保存最高有效 30 位（节省 6.25%）。在指令缓存重填时，遇到的任何低位清零的指令应在存储在缓存中之前重新编码为非法 30 位指令，以保持非法指令异常行为。
>
> 更重要的是，通过将我们的基础 ISA 压缩到 32 位指令字的子集中，我们为非标准和定制扩展留下了更多可用空间。特别是，基础 RV32I ISA 使用 32 位指令字中少于 1/8 的编码空间。不需要支持标准压缩指令扩展的实现可以将 3 个额外的不符合标准 30 位指令空间映射到 32 位固定宽度格式中，同时保持对标准 ≥32 位指令集扩展的支持。

位 [15:0] 全为零的编码被定义为非法指令。这些指令被认为是最小长度：如果存在任何 16 位指令集扩展则为 16 位，否则为 32 位。位 [ILEN-1:0] 全为一的编码也是非法的；这个指令被认为是 ILEN 位长。

> **注释：**
> 我们认为包含所有零位的任何长度指令都不合法是一个特性，因为这快速捕获错误跳转到归零内存区域。类似地，我们也保留包含所有一的指令编码为非法指令，以捕获与未编程非易失性内存设备、断开的内存总线或损坏的内存设备观察到的其他常见模式。
>
> 软件可以依赖包含零的自然对齐 32 位字在所有 RISC-V 实现上作为非法指令，供软件在明确需要非法指令时使用。由于变长编码，为所有一定义相应的已知非法值更困难。软件通常不能使用 ILEN 位全为 1 的非法值，因为软件可能不知道最终目标机器的 ILEN（例如，如果软件编译为许多不同机器使用的标准二进制库）。也考虑将所有一的 32 位字定义为非法，因为所有机器必须支持 32 位指令大小，但这要求 ILEN >32 的机器上的指令获取单元在这种指令与保护边界接壤时报告非法指令异常而不是访问故障异常，使变指令长度获取和解码复杂化。

RISC-V 基础 ISA 具有小端或大端内存系统，特权架构进一步定义了双端操作。指令作为 16 位小端包裹序列存储在内存中，无论内存系统字节序如何。形成一个指令的包裹存储在递增的半字地址，最低地址包裹保存指令规范中最低编号的位。

> **注释：**
> 我们最初为 RISC-V 内存系统选择小端字节排序，因为小端系统目前在商业上占主导地位（所有 x86 系统；ARM 的 iOS、Android 和 Windows）。一个小点是我们也发现小端内存系统对硬件设计师更自然。然而，某些应用领域，如 IP 网络，操作大端数据结构，某些遗留代码库已建立假设大端处理器，因此我们定义了 RISC-V 的大端和双端变体。
>
> 我们必须固定指令包裹在内存中存储的顺序，独立于内存系统字节序，以确保长度编码位总是首先以半字地址顺序出现。这允许指令获取单元通过仅检查第一个 16 位指令包裹的前几位快速确定变长指令的长度。
>
> 我们进一步使指令包裹本身为小端，完全将指令编码与内存系统字节序解耦。这种设计对软件工具和双端硬件都有益。否则，例如，RISC-V 汇编器或反汇编器总是需要知道预期的活动字节序，尽管在双端系统中，字节序模式可能在执行期间动态改变。相反，通过给指令固定字节序，有时可能使仔细编写的软件即使在二进制形式中也与字节序无关，很像位置无关代码。
>
> 然而，使指令仅为小端的选择对编码或解码机器指令的 RISC-V 软件确实有后果。例如，大端 JIT 编译器在存储到指令内存时必须交换字节顺序。
>
> 一旦我们决定固定小端指令编码，这自然导致将长度编码位放在指令格式的 LSB 位置以避免破坏操作码字段。

## 异常、陷阱和中断

我们使用术语**异常**来指代在运行时发生的与当前 RISC-V 硬件线程中指令相关的异常情况。我们使用术语**中断**来指代可能导致 RISC-V 硬件线程经历意外控制转移的外部异步事件。我们使用术语**陷阱**来指代由异常或中断引起的控制转移到陷阱处理程序。

以下章节中的指令描述描述了在执行期间可能引发异常的条件。大多数 RISC-V EEI 的一般行为是当在指令上发出异常信号时发生到某个处理程序的陷阱（除了浮点异常，在标准浮点扩展中不引起陷阱）。中断如何生成、路由到硬件线程以及由硬件线程启用取决于 EEI。

> **注释：**
> 我们对"异常"和"陷阱"的使用与 IEEE-754 浮点标准兼容。

陷阱如何处理并对硬件线程上运行的软件可见取决于封闭执行环境。从在执行环境内运行的软件的角度来看，硬件线程在运行时遇到的陷阱可能有四种不同效果：

包含陷阱:::
  陷阱对在执行环境内运行的软件可见并由其处理。例如，在为硬件线程提供监管者和用户模式的 EEI 中，用户模式硬件线程的 ECALL 通常导致控制转移到在同一硬件线程上运行的监管者模式处理程序。类似地，在同一环境中，当硬件线程被中断时，中断处理程序将在硬件线程上以监管者模式运行。

请求陷阱:::
  陷阱是同步异常，是对执行环境的显式调用，代表执行环境内的软件请求行动。示例是系统调用。在这种情况下，执行环境采取请求的行动后，执行可能在硬件线程上恢复或不恢复。例如，系统调用可能删除硬件线程或导致整个执行环境的有序终止。

不可见陷阱:::
  陷阱由执行环境透明处理，陷阱处理后执行正常恢复。示例包括仿真缺失指令、在需求分页虚拟内存系统中处理非驻留页面故障，或在多程序机器中为不同作业处理设备中断。在这些情况下，在执行环境内运行的软件不知道陷阱（我们在这些定义中忽略时间效果）。

致命陷阱:::
  陷阱表示致命故障并导致执行环境终止执行。示例包括虚拟内存页面保护检查失败或允许看门狗定时器到期。每个 EEI 应定义如何终止执行并报告给外部环境。

下表显示了每种陷阱的特征。

.陷阱特征
[%autowidth,float="center",align="center",cols="<,^,^,^,^",options="header",]
|===
| |包含 |请求 |不可见 |致命
|执行终止 |否 |否^1^|否 |是
|软件遗忘 |否 |否 |是 |是^2^
|由环境处理 |否 |是 |是 |是
|===

^1^ 可能请求终止 +
^2^ 不精确致命陷阱可能被软件观察到

EEI 为每个陷阱定义是否精确处理，尽管建议在可能的情况下保持精确性。包含和请求陷阱可以被执行环境内的软件观察为不精确。根据定义，不可见陷阱不能被在执行环境内运行的软件观察为精确或不精确。如果已知错误指令不导致立即终止，则致命陷阱可以被在执行环境内运行的软件观察为不精确。

因为本文档描述非特权指令，很少提及陷阱。处理包含陷阱的架构手段在特权架构手册中定义，以及支持更丰富 EEI 的其他特性。仅定义为引起请求陷阱的非特权指令在此记录。不可见陷阱根据其性质超出本文档范围。此处未定义且未由其他手段定义的指令编码可能引起致命陷阱。

## 未指定的行为和值

架构完全描述了实现必须做什么以及它们可能做什么的任何约束。在架构有意不约束实现的情况下，明确使用术语 UNSPECIFIED。

术语 UNSPECIFIED 指有意不受约束的行为或值。这些行为或值的定义对扩展、平台标准或实现开放。扩展、平台标准或实现文档可能提供规范内容以进一步约束基础架构定义为 UNSPECIFIED 的情况。

与基础架构一样，扩展应完全描述允许的行为和值，并对有意不受约束的情况使用术语 UNSPECIFIED。这些情况可能被其他扩展、平台标准或实现约束或定义。
